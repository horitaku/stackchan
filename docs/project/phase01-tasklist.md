# フェーズ 1 タスクリスト（プロトコル v0）

## 1. このドキュメントの目的

フェーズ 1（プロトコル v0）を実行しやすくするために、作業を具体タスクへ分解して管理します。
本ドキュメントは「日次で更新する実行用リスト」です。

## 2. 運用ルール

- ステータスは `Planned`、`In Progress`、`Blocked`、`Done` を使用します。
- 作業開始時に `開始日` を記録し、完了時に `完了日` を記録します。
- `Blocked` になった場合は、必ず `ブロック理由` と `解除条件` を記載します。
- 1 タスクは原則 0.5 日から 2 日で終わる粒度に保ちます。

## 3. フェーズ 1 完了条件

次のすべてを満たしたらフェーズ 1 完了とします。

- WebSocket エンベロープ（type、timestamp、session_id、sequence、version）が定義済みである。
- 最小イベントセット（session.hello、session.welcome、error、audio.chunk、audio.end）が文書化済みである。
- JSON Schema とペイロード例が揃っている。
- 互換性ルール（後方互換、追加方針、非推奨化方針）が明文化されている。
- サーバー実装着手に必要な前提と未解決事項が記録されている。

## 5. タスク詳細（実行手順）

### P1-01 エンベロープ共通項目を定義する

- 作業内容
  - 共通フィールドの意味、型、必須/任意、制約を定義します。
- 完了条件
  - すべてのイベントが同一エンベロープ規則で表現できる状態です。
- 確認観点
  - server と firmware の双方で解釈可能な定義になっていること。

### P1-02 最小イベントセットを定義する

- 作業内容
  - 5 イベントの用途、入力、出力、異常系を定義します。
- 完了条件
  - イベント間の遷移と前後関係が説明できる状態です。
- 確認観点
  - セッション開始から音声終端までの最小シナリオが表現できること。

### P1-03 JSON Schema を作成する

- 作業内容
  - 各イベントと共通エンベロープのスキーマを作成します。
- 完了条件
  - 必須項目、型、追加プロパティ方針が明示されています。
- 確認観点
  - サンプルペイロードがスキーマに適合すること。

### P1-04 ペイロード例を作成する

- 作業内容
  - 正常系と代表的な異常系の JSON 例を作成します。
- 完了条件
  - 実装者が迷わず送受信データを組み立てられる状態です。
- 確認観点
  - sequence、timestamp、session_id の扱いが一貫していること。

### P1-05 バージョニングと互換性ルールを整理する

- 作業内容
  - version フィールド運用、非互換変更時の扱い、追加方針を定義します。
- 完了条件
  - 将来拡張時の判断基準が文書化されています。
- 確認観点
  - 「追加的進化」の原則に沿ったルールになっていること。

### P1-06 プロトコル検証観点を定義する

- 作業内容
  - 検証項目を整理し、フェーズ 2 以降で使えるチェックリストを作成します。
- 完了条件
  - 実装前に最低限チェックすべき観点が揃っています。
- 確認観点
  - フィールド欠落、型不正、順序不整合、異常終了の観点を含むこと。

### P1-07 フェーズ 2 引き継ぎ事項を整理する

- 作業内容
  - サーバースケルトン実装に必要な前提と未決事項を整理します。
- 完了条件
  - フェーズ 2 着手時のブロッカーが明確です。
- 確認観点
  - セッション管理、ログ相関、エラー通知の前提が明記されていること。

### P1-08 フェーズ 1 完了レビューを実施する

- 作業内容
  - 完了条件の満たし込み確認を行います。
  - 未解決事項を次フェーズへ引き継ぎます。
- 完了条件
  - 合意済みのレビュー結果が記録されています。
- 確認観点
  - フェーズ 2 の作業開始判断が可能であること。

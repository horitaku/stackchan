# フェーズ 3 タスクリスト（Provider 境界）

## 1. このドキュメントの目的

フェーズ 3（Provider 境界）を実行しやすくするために、作業を具体タスクへ分解して管理します。
本ドキュメントは「日次で更新する実行用リスト」です。

## 2. 運用ルール

- ステータスは `Planned`、`In Progress`、`Blocked`、`Done` を使用します。
- 作業開始時に `開始日` を記録し、完了時に `完了日` を記録します。
- `Blocked` になった場合は、必ず `ブロック理由` と `解除条件` を記載します。
- 1 タスクは原則 0.5 日から 2 日で終わる粒度に保ちます。

## 3. フェーズ 3 完了条件

次のすべてを満たしたらフェーズ 3 完了とします。

- STT、LLM、TTS のインターフェースが server 側で定義済みである。
- Provider 実装を差し替え可能な依存注入構成が導入されている。
- mock Provider 実装（STT、LLM、TTS）が最小シナリオを動作可能にしている。
- Provider 呼び出し境界でタイムアウト、キャンセル、リトライ方針が明示されている。
- Provider エラーが protocol error へマッピングされ、ログに相関情報が残る。
- hello/welcome 以降の最小会話オーケストレーションが自動テストで検証されている。
- フェーズ 4（最小音声パス）へ引き継ぐ前提と未解決事項が記録されている。

## 5. タスク詳細（実行手順）

### P3-01 Provider 境界の責務を定義する

- 作業内容
  - conversation / session 層と providers 層の境界を明文化します。
  - Provider が扱う入出力形式（テキスト、音声メタデータ、エラー）を整理します。
- 完了条件
  - 実装者が provider 層に含める責務と含めない責務を判断できます。
- 確認観点
  - server 本体ロジックがベンダー SDK へ直接依存しない方針が明示されていること。

### P3-02 STT、LLM、TTS インターフェースを定義する

- 作業内容
  - STT、LLM、TTS の最小インターフェース（入力、出力、エラー）を定義します。
  - context.Context を引数に含め、キャンセルを透過させます。
- 完了条件
  - 単体テストで mock 実装へ差し替え可能な型定義が揃っています。
- 確認観点
  - 返却型が protocol と接続可能であること。

### P3-03 Provider 設定と依存注入を実装する

- 作業内容
  - 起動時に Provider を組み立てる初期化コードを追加します。
  - 将来の OpenAI / Voicevox 実装に置き換え可能な DI 構成を導入します。
- 完了条件
  - server 起動時に mock Provider を注入して稼働できます。
- 確認観点
  - main パッケージ以外に具体実装の new 呼び出しが漏れていないこと。

### P3-04 mock STT Provider を実装する

- 作業内容
  - 固定入力に対して決定的な文字列を返す mock STT を実装します。
  - タイムアウト確認用に遅延シミュレーションを追加します。
- 完了条件
  - テストから安定して同一結果を得られます。
- 確認観点
  - context cancel 時に即時終了すること。

### P3-05 mock LLM Provider を実装する

- 作業内容
  - 入力テキストを受け、ルールベースの定型応答を返す mock LLM を実装します。
  - 入力検証エラーと一時エラーを再現できるようにします。
- 完了条件
  - 正常系と異常系をテストで再現可能です。
- 確認観点
  - 冪等性が必要な呼び出しで結果が安定していること。

### P3-06 mock TTS Provider を実装する

- 作業内容
  - テキスト入力から疑似音声データのメタ情報（長さ、サンプルレート等）を返します。
  - 将来の音声バイナリ送信に備えた構造体を定義します。
- 完了条件
  - フェーズ 4 で audio.chunk へ接続できる出力形式が確認できます。
- 確認観点
  - 音声フォーマット前提（sample rate、frame 長）がコード上で明示されていること。

### P3-07 Provider 呼び出しポリシーを実装する

- 作業内容
  - timeout、最大試行回数、指数バックオフ方針を境界レイヤーで統一します。
  - 非 retryable エラーを即時返却する分岐を実装します。
- 完了条件
  - Provider ごとに異なる失敗挙動を統一的に扱えます。
- 確認観点
  - 無限リトライが発生しないこと。

### P3-08 Provider エラー変換を実装する

- 作業内容
  - provider 内部エラーを protocol error コードへマッピングします。
  - ログへ session_id、request_id、provider 名を記録します。
- 完了条件
  - クライアントに返す error イベントが一貫した形式になります。
- 確認観点

### P3-09 最小会話オーケストレーションを実装する

## 6. 前提と設計メモ

### 採用技術候補

| 用途 | 候補 | 備考 |
| --- | --- | --- |
| Provider 境界 | `server/internal/providers` | interface-first で実装 |
| リトライ制御 | 自前実装または `cenkalti/backoff/v4` | 指数バックオフを優先 |
| タイムアウト管理 | `context.WithTimeout` | 呼び出し単位で制御 |
| テストダブル | mock struct / fake 実装 | 決定的挙動を重視 |

### 最小オーケストレーション状態遷移

```text
Connected 状態
  └── 入力受付（テストハーネス）
        └── STT 呼び出し
              └── LLM 呼び出し
                    └── TTS 呼び出し
                          ├── 成功: 応答イベント送信
                          └── 失敗: error イベント送信
```

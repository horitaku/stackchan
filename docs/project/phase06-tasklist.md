# フェーズ 6 タスクリスト（再生とアバター同期）

## 1. このドキュメントの目的

フェーズ 6（再生とアバター同期）を実行しやすくするために、作業を具体タスクへ分解して管理します。
本ドキュメントは「日次で更新する実行用リスト」です。

## 2. 運用ルール

- ステータスは `Planned`、`In Progress`、`Blocked`、`Done` を使用します。
- 作業開始時に `開始日` を記録し、完了時に `完了日` を記録します。
- `Blocked` になった場合は、必ず `ブロック理由` と `解除条件` を記載します。
- 1 タスクは原則 0.5 日から 2 日で終わる粒度に保ちます。

## 3. フェーズ 6 完了条件

次のすべてを満たしたらフェーズ 6 完了とします。

- firmware が `tts.end` を受信した際に `audio_base64` をデコードし、M5Stack CoreS3 のスピーカーで再生できる。
- 音声再生中にリップシンク用の口形状パラメータ（最低限: 振幅ベース）が更新される。
- `avatar.expression` / `motion.play` を受信して、アバター表情と最小モーション状態が反映される。
- 再生状態（開始、進行中、終了、失敗）が firmware 内で状態遷移として管理される。
- 再生遅延、再生時間、再生エラーが構造化ログまたはシリアルログで追跡できる。
- 再生中断（切断、再接続、新規リクエスト割り込み）時に安全に停止し、次の再生へ復帰できる。
- server 側で再生制御に必要なイベント定義が protocol と整合し、必要な schema/example が更新されている。
- 再生とアバター同期の最小エンドツーエンド確認手順が `firmware/README.md` に記録されている。
- フェーズ 7（WebUI と可観測性）へ引き継ぐ監視項目と未解決事項が記録されている。

## 5. タスク詳細（実行手順）

### P6-01 再生パイプラインの責務境界を定義する

- 作業内容
  - firmware 内の責務を「デコード」「再生制御」「アバター同期」「イベント受信」に分離して定義します。
  - server 側責務（音声生成、イベント送信）との境界を明文化します。
  - 再生状態遷移（Idle/Buffering/Playing/Stopping/Error）を設計します。
- 完了条件
  - 実装者が再生処理と同期処理の配置先を迷わず判断できる状態です。
- 確認観点
  - firmware に AI オーケストレーション責務が入っていないこと。

### P6-02 `tts.end` ペイロードのデコード実装を追加する

- 作業内容
  - `tts.end.payload.audio_base64` をデコードする処理を実装します。
  - サイズ上限、空データ、不正 Base64 を検知しエラー処理します。
  - `request_id` 単位でデコード結果を再生キューへ渡します。
- 完了条件
  - 正常な `audio_base64` をバイト列へ変換し、再生モジュールへ受け渡せます。
- 確認観点
  - デコード失敗時にセッションを壊さず `error` ログを記録できること。

### P6-03 CoreS3 スピーカー再生モジュールを実装する

- 作業内容
  - `firmware/runtime/audio/` に再生モジュールを追加し、PCM 再生の最小実装を行います。
  - sample rate、channel、bit depth を `tts.end` メタデータから設定します。
  - 再生完了コールバックを提供し、後続の状態遷移へ接続します。
- 完了条件
  - デコード済み音声を CoreS3 スピーカーで再生できます。
- 確認観点
  - 再生中のバッファアンダーランや過大バッファ時に安全に失敗処理できること。

### P6-04 再生状態管理と割り込み制御を実装する

- 作業内容
  - 再生状態マシン（Idle/Buffering/Playing/Stopping/Error）を導入します。
  - 新規 `request_id` 到着時の割り込みポリシー（停止して切替 or キュー積み）を定義します。
  - WebSocket 切断時や再接続時に再生を安全停止します。
- 完了条件
  - 状態不整合や二重再生なしで、再生開始から停止まで一貫して管理できます。
- 確認観点
  - 再接続後に次の再生要求を正常に処理できること。

### P6-05 リップシンク抽象レイヤを実装する

### 再生・同期状態遷移（初期案）

```text
Idle
  └── tts.end 受信
        └── Decode（成功）
              └── Buffering
                    └── Playing
                          ├── 再生完了 -> Idle
                          ├── 新規 request 到着 -> Stopping -> Buffering
                          └── 出力失敗 -> Error -> Idle
        └── Decode（失敗） -> Error -> Idle
```

### メトリクス観測項目（初期案）

- `playback_start_latency_ms`（`tts.end` 受信から再生開始まで）
- `playback_duration_ms`（再生実時間）
- `lip_sync_update_interval_ms`（口形状更新周期）
- `playback_error_count`（デコード失敗、出力失敗、割り込み失敗）
